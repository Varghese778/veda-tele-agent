# 📋 OreHack — README Submission Guidelines

Your GitHub repository **must include a README.md** that follows this structure. The evaluation engine reads your README as part of scoring — a well-written README directly improves your Documentation and Problem Alignment scores.

---

## Required Sections

Your README must include all of the following sections in this order:

### 1. Project Title & Tagline
One-line description of what your project does. Be specific — avoid vague phrases like "an innovative solution."

### 2. Problem Statement
In your own words, explain the problem your project solves. Reference the problem statement given to you. This should be 3–5 sentences.

### 3. Features
A table or bullet list of what your project actually does. Each feature should name the functionality and describe what it does in one sentence. Do not list things you planned but did not build.

### 4. Tech Stack
List every technology, framework, and library your project uses. Separate frontend and backend clearly.

### 5. Project Structure
A directory tree of your repository with a short comment on what each major file/folder does. This is one of the most important sections for scoring — it shows code organization.

### 6. Installation & Setup
Step-by-step instructions to run your project locally. Include every command needed from clone to running. Assume the reader has never seen your project before.

### 7. How It Works
Explain the actual technical flow of your application — not what it does, but *how* it does it. Walk through each major feature as a numbered flow (input → processing → output).

### 8. Scalability
Explain how your project could scale. Address: concurrent users, data volume, deployment (Docker/cloud), and any bottlenecks you identified. Minimum 3 points.

### 9. Feasibility
Explain why this project is practically buildable and deployable. Address: tools used, dependencies, infrastructure requirements, and what would be needed to take it to production.

### 10. Novelty
Explain what makes your approach different from existing tools or solutions. Be honest — do not claim novelty that doesn't exist. Explain the specific gap you are filling.

### 11. Feature Depth
Explain the depth of your key features beyond surface-level. What edge cases do you handle? What configurations are supported? What data does the system produce?

### 12. Ethical Use & Disclaimer *(if applicable)*
Required for any project involving security scanning, data collection, user tracking, or AI-generated content.

### 13. License
Include a license. Apache 2.0 or MIT recommended.

### 14. Author
Your name, email, and GitHub link.

---

## Rules

- **Write only what you actually built.** The evaluator checks your repository — claims not backed by code will hurt your score.
- **No placeholder text.** Every section must have real content specific to your project.
- **Keep it precise.** One clear sentence beats three vague ones.
- **English only.**
- **No copy-paste from the problem statement** without adding your own explanation of how you addressed it.

---

## Scoring Impact

| README Section | Affects Score Category |
|---|---|
| Problem Statement | Problem Alignment |
| Features + Feature Depth | Functionality, Product Features |
| Tech Stack | Tech Stack Efficiency |
| Project Structure | Code Quality & Modularity |
| How It Works | Architecture, Documentation |
| Scalability | Scalability |
| Feasibility | Feasibility |
| Novelty | Novelty |
| Installation & Setup | Deployment Readiness, Project Polish |

A complete, honest, well-written README can add up to **15 points** to your final score compared to a missing or vague one.
